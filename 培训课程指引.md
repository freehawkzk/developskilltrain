# 实验室开发技能培训课程 —— 课程构建指引

> 面向：CI/助听器研究实验室本科生与硕士研究生
> 编写视角：博导 + 企业培训导师
> 核心目标：让学生从"开发零基础"到"能读懂、能跑通、能改动手中的研究代码"

---

## 一、总体设计哲学

### 1.1 学情分析

你的学生群体有三个关键特征，决定了课程不能照搬计算机学院的深度学习课程：

**物理/光电背景，数学基础好但编程思维弱。** 他们理解卷积的数学定义，但不知道"卷积层在代码里长什么样"。你的课程要反复把"数学公式 → 代码实现"这条线画清楚，这是他们最容易产生顿悟的桥梁。

**研究导向，不是工程导向。** 他们学深度学习是为了读懂论文、复现实验、修改模型，不是为了上生产。所以课程重心应放在"理解架构、读懂代码、调整实验"，而不是"模型部署、工程优化"。

**经验跨度大。** 有的本科生可能连命令行都没摸过，有的硕士生可能已经用过 Python。每次课都要有分层设计——基础路径保证所有人跟上，进阶任务让有经验的学生不无聊。

### 1.2 五条核心原则

**原则一：领域锚定。** 每一个技术概念都要用声音/CI场景来引入和练习。不要用 MNIST 识别手写数字——用 ESC-50 识别环境声音；不要用 ImageNet 分类图片——用语音帧分类清浊音。学生在自己研究的语境中学东西，注意力、记忆深度、迁移能力都显著更强。

**原则二：先动手后理解（Doing Before Knowing）。** 企业培训中反复验证的原则：先让学生把代码跑通、看到结果、获得成就感，然后再讲背后的原理。顺序是"现象 → 问题 → 原理"，而不是"原理 → 公式 → 代码"。这跟传统课堂教学正好反过来，但对零基础成人学习者效果好得多。

**原则三：螺旋上升。** 同一个概念（如卷积、损失函数、梯度下降）在不同模块中以不同深度反复出现。第一轮只要求"知道这是啥"，第二轮要求"能调参数看效果"，第三轮要求"能修改结构"。

**原则四：渐进式项目。** 从第一节课开始就有一个贯穿全课程的"主线项目"（例如：构建一个CI语音增强pipeline），每节课完成后，主线项目就往前推进一步。这给学生持续的目标感和成就感，也让知识之间产生连接，而不是各自孤立。

**原则五：降低挫败感阈值。** 环境配置是培训中的头号杀手。环境搭不好，后面的课全部废掉。第一模块必须把这个问题彻底解决，宁可多花一节课。给学生提供预配置的 Docker 镜像或 Conda environment.yml，让他们"一键启动"。

### 1.3 贯穿主线项目

建议设定以下主线项目，随课程推进逐步完成：

**项目名称：CI语音增强与识别 Pipeline**

- 模块1完成后：能在Linux上运行Python脚本，环境就绪
- 模块2完成后：能训练一个简单的神经网络做语音/噪声二分类
- 模块3完成后：能搭建声音分类模型，区分不同类型的环境声
- 模块4完成后：能运行DeepACE，理解CI编码策略与深度学习的结合方式
- 模块5完成后：能用DeepFilterNet做语音增强，并评估增强效果
- 模块6完成后：能将增强后的语音送入ASR，评估端到端的识别效果

最终，每个学生手里都有一个"从原始音频 → 语音增强 → CI编码策略 → 语音识别"的完整 pipeline，这直接对接实验室的研究工作。

## 二、模块详细设计

### 模块1：Linux基本使用与深度学习环境搭建

**课时：2次课 × 2.5小时 = 5小时**

这是整个培训的地基。地基不牢，后续全部动摇。宁可这里慢，也不要这里赶。

#### 第1次课：Linux生存指南（2.5小时）

**教学目标：** 学生能在Linux终端下完成基本的文件操作、进程管理、远程连接，不再对黑框框恐惧。

**内容与节奏：**

| 时间 | 内容 | 形式 | 要点 |
|------|------|------|------|
| 0:00-0:20 | 为什么要用Linux？服务器场景介绍 | 讲授 | 用类比："你的电脑是工作站，服务器是工厂" |
| 0:20-0:50 | 文件系统与基本命令：cd, ls, pwd, mkdir, rm, cp, mv, cat, less | 边讲边练 | 每个命令讲完立即让学生跟着做 |
| 0:50-1:00 | 休息 | — | — |
| 1:00-1:30 | 权限、管道、重定向、grep, find | 边讲边练 | 用声音处理的例子："找出目录下所有.wav文件" |
| 1:30-1:50 | SSH远程连接、SCP文件传输、tmux基本使用 | 演示+实操 | 这是他们之后每天都要用的 |
| 1:50-2:20 | Vim基础操作 + VS Code Remote | 演示+实操 | 不要试图教会Vim，教到能改配置文件就够 |
| 2:20-2:30 | 小挑战：连接服务器，找到指定wav文件，查看其大小 | 实操 | 完成即达标 |

**关键决策说明：**

为什么讲 tmux？因为学生之后在服务器上跑训练，必须知道如何让程序在后台运行、断开连接不中断。这是实验室最常见的痛点之一。

为什么讲 VS Code Remote 而不是只讲 Vim？因为物理背景的学生对 IDE 更熟悉，VS Code Remote 的门槛远低于 Vim，能大幅降低他们的抵触感。Vim 只教到"能改配置文件"即可。

**课前准备：**
- 提前为每个学生分配服务器账号
- 准备一份Linux速查表（一页纸，打印+电子版）
- 在服务器上准备好示例音频文件目录

**课后任务（分层）：**
- 基础：完成一份"在服务器上找到并播放一个wav文件"的操作记录
- 进阶：写一个bash脚本，批量转换目录下所有wav文件的采样率

#### 第2次课：深度学习环境搭建（2.5小时）

**教学目标：** 每个学生都能在自己的环境里 `import torch` 并成功调用GPU，能使用 Conda 管理依赖。

**内容与节奏：**

| 时间 | 内容 | 形式 | 要点 |
|------|------|------|------|
| 0:00-0:15 | 回顾上节课 + 环境问题答疑 | 互动 | 上节课一定会有遗留问题 |
| 0:15-0:45 | Conda环境管理：创建、激活、安装包、导出环境 | 边讲边练 | 强调"一个项目一个环境"的习惯 |
| 0:45-1:05 | pip vs conda、requirements.txt、environment.yml | 讲授+演示 | 用通俗类比："环境就是独立的工具箱" |
| 1:05-1:15 | 休息 | — | — |
| 1:15-1:35 | PyTorch安装、CUDA验证 | 边讲边练 | 提前准备好安装命令，避免现场查版本 |
| 1:35-1:55 | Docker基础概念 + 预配置镜像使用 | 演示 | 不要求会写Dockerfile，只要会用 |
| 1:55-2:20 | Jupyter Lab配置与远程访问 | 边讲边练 | 之后所有课程都用Jupyter |
| 2:20-2:30 | 验证环节：import torch, torch.cuda.is_available() | 实操 | 全员绿灯才算通过 |

**关键决策说明：**

为什么用 Conda 而不是 venv？因为学生的依赖里经常有 numpy/scipy/librosa 这类有 C 扩展的库，Conda 的预编译二进制包能避免大量编译问题。等他们有经验了再介绍 venv + pip。

为什么提供 Docker 镜像？因为一定会有学生在自己的笔记本上装不上 CUDA。Docker 是"环境不行"的终极兜底方案。镜像里预装好 PyTorch + librosa + torchaudio + jupyterlab，学生只需要 `docker run`。

**课前准备：**
- 准备 Conda environment.yml 文件
- 准备 Docker 镜像并推送到实验室镜像仓库
- 准备一份"环境搭建完整步骤"文档，确保每个命令都经过验证
- 准备一个"环境问题排查清单"（常见报错及解决方案）

**课后任务（分层）：**
- 基础：在服务器上创建新环境，安装PyTorch，运行一个简单的张量运算
- 进阶：用Docker启动预配置镜像，对比两种方式的差异

### 模块2：深度学习入门

**课时：3次课 × 2.5小时 = 7.5小时**

这个模块是"理解"的分水岭。学生要么在这里建立对深度学习的直觉，要么在这里开始掉队。关键是用声音的例子讲，不是用图像的例子讲。

#### 第1次课：从感知机到神经网络（2.5小时）

**教学目标：** 理解"学习"就是"调参数"，理解反向传播的直觉（不需要推导公式），理解神经网络的训练循环。

**内容与节奏：**

| 时间 | 内容 | 形式 | 要点 |
|------|------|------|------|
| 0:00-0:30 | 什么是"学习"？从线性回归开始 | 讲授+动画 | 用纯Python实现线性回归，展示"逐步逼近"的过程 |
| 0:30-0:55 | 从线性到非线性：激活函数的必要性 | 讲授+可视化 | 用"声音响度感知"的例子——人对响度的感知不是线性的，这正是激活函数的物理直觉 |
| 0:55-1:10 | 休息 | — | — |
| 1:10-1:40 | 多层感知机(MLP) + 反向传播直觉 | 讲授+动画 | 不推导链式法则，用"误差信号回传"的直觉讲。比喻："每个参数都在问——我该变大还是变小？" |
| 1:40-2:10 | PyTorch基础：张量、自动微分、nn.Module | 边讲边练 | 把刚才的MLP用PyTorch重写，展示"同一件事，PyTorch帮你做了什么" |
| 2:10-2:30 | 实操：训练MLP对两个正弦波进行分类 | 实操 | 第一个真正的训练任务！确保每个人都能看到loss下降 |

**关键设计：** 第一个训练任务极其重要。它要简单到不会出错，但又要能看到"模型在学习"的过程。对两个不同频率的正弦波分类是完美选择——输入是波形的一帧，输出是"低频"或"高频"，数据可以代码生成无需下载，模型2-3层MLP就够了，训练10秒就能收敛。

#### 第2次课：卷积神经网络与声音的"图像化"（2.5小时）

**教学目标：** 理解为什么音频可以变成"图像"来处理，理解卷积操作在频谱图上的含义，能搭建并训练一个CNN。

**内容与节奏：**

| 时间 | 内容 | 形式 | 要点 |
|------|------|------|------|
| 0:00-0:20 | 回顾 + 上节课任务答疑 | 互动 | — |
| 0:20-0:50 | 音频基础：采样率、频谱、语谱图(spectrogram) | 讲授+可视化 | 这是整个音频深度学习的基石。用librosa实时展示同一段语音的不同表示 |
| 0:50-1:10 | 为什么语谱图是"图像"？→ CNN可以处理声音 | 讲授 | 关键类比："语谱图横轴是时间、纵轴是频率、颜色是能量——CNN天生擅长这种2D模式识别" |
| 1:10-1:20 | 休息 | — | — |
| 1:20-1:50 | CNN核心概念：卷积、池化、步长、填充 | 边讲边练 | 用语谱图的例子："一个3×3的卷积核在看一小块时频区域，寻找某种模式" |
| 1:50-2:20 | PyTorch实现CNN：从nn.Conv2d到完整分类网络 | 边讲边练 | 搭建一个3层CNN，对语谱图做分类 |
| 2:20-2:30 | 实操：训练CNN对语音vs噪声进行分类 | 实操 | 主线项目第一阶段 |

**关键设计：** 语谱图是音频深度学习最重要的"枢纽概念"——几乎所有音频模型（包括DeepACE和DeepFilterNet）都在某种时频表示上工作。这节课要把这个概念彻底讲透。建议准备多段音频的可视化对比：纯净语音、噪声、语音+噪声的语谱图，让学生直观看到"增强就是把这个图变干净"。

#### 第3次课：训练技巧与实验方法论（2.5小时）

**教学目标：** 知道怎么评估模型、怎么调参、怎么做实验记录。这是从"能跑通"到"能做研究"的关键一步。

**内容与节奏：**

| 时间 | 内容 | 形式 | 要点 |
|------|------|------|------|
| 0:00-0:15 | 回顾 + 上节课任务答疑 | 互动 | — |
| 0:15-0:40 | 过拟合/欠拟合、训练/验证/测试集划分、交叉验证 | 讲授+演示 | 用上节课的CNN实际演示过拟合——把训练集缩小到10个样本，看训练loss下降但验证loss上升 |
| 0:40-1:05 | 学习率、批大小、优化器选择、学习率调度 | 讲授+实验 | 在Jupyter里现场改学习率，让学生看"太大→震荡、太小→不动" |
| 1:05-1:15 | 休息 | — | — |
| 1:15-1:40 | 数据增强（尤其是音频领域的增强：时间拉伸、频谱掩蔽、加噪） | 讲授+演示 | 这些增强方法直接对接后面的DeepFilterNet |
| 1:40-2:10 | 实验管理：TensorBoard/Weights & Biases、实验记录习惯 | 边讲边练 | 强调："不做记录的实验等于没做" |
| 2:10-2:30 | 实操：改进主线项目的CNN——加入数据增强、调整学习率、用TensorBoard记录 | 实操 | — |

**关键设计：** 这节课最容易被忽视，但对研究来说最重要。很多学生训练完模型就算了，不记录超参数、不保存checkpoint、不做对比实验，后面写论文时完全无法复现。在课上就要建立"每次实验都记录"的习惯。

**模块2课后综合任务（分层）：**
- 基础：完成主线项目第一阶段——训练一个能区分语音和噪声的CNN，准确率 > 85%
- 进阶：尝试不同的网络结构（加深、加宽、加ResNet连接），在TensorBoard中对比结果

### 模块3：面向声音的分类模型

**课时：3次课 × 2.5小时 = 7.5小时**

这个模块是"从通用DL到音频DL"的桥梁，也是后续两个模型解析课的基础。

#### 第1次课：音频特征工程与经典方法（2.5小时）

**教学目标：** 理解音频信号从原始波形到模型输入的完整转换流程，掌握MFCC、梅尔语谱图等关键特征。

**内容与节奏：**

| 时间 | 内容 | 形式 | 要点 |
|------|------|------|------|
| 0:00-0:20 | 回顾 + 模块2任务点评 | 互动 | — |
| 0:20-0:55 | 音频特征全景：从波形到MFCC | 讲授+可视化 | 画出完整流程：预加重→分帧→加窗→FFT→梅尔滤波→对数→DCT→MFCC。每一步都在Jupyter里可视化 |
| 0:55-1:10 | 为什么是梅尔刻度？——从CI的频率映射讲起 | 讲授 | 这是跟实验室研究最直接的连接点：CI的电极大致对应梅尔频率带，所以梅尔语谱图天然适合CI研究 |
| 1:10-1:20 | 休息 | — | — |
| 1:20-1:50 | MFCC vs 梅尔语谱图 vs 原始波形：不同特征的特点与适用场景 | 对比演示 | 用同一个分类任务，对比三种特征的效果 |
| 1:50-2:20 | torchaudio实战：音频加载、变换、特征提取 | 边讲边练 | torchaudio.transforms 是之后所有模型代码的基础 |
| 2:20-2:30 | 实操：对ESC-50数据集提取梅尔语谱图特征 | 实操 | 为下节课做准备 |

**关键设计：** 梅尔刻度与CI电极映射的关系是这个模块的"金钥匙"。这不是一个巧合——CI的电极从底转到顶转，频率分配本身就和梅尔刻度高度相关。让学生在这里建立这个直觉，后面理解DeepACE的频率通道选择就水到渠成了。

#### 第2次课：音频分类模型搭建（2.5小时）

**教学目标：** 能从零搭建一个音频分类pipeline（数据加载 → 特征提取 → 模型定义 → 训练 → 评估），理解不同架构的选择。

**内容与节奏：**

| 时间 | 内容 | 形式 | 要点 |
|------|------|------|------|
| 0:00-0:15 | 回顾 + 上节课任务答疑 | 互动 | — |
| 0:15-0:45 | PyTorch Dataset/DataLoader for Audio | 边讲边练 | 这是学生最容易卡住的地方——"音频数据怎么喂给模型？" |
| 0:45-1:10 | 模型架构选择：CNN → CRNN → Attention-based | 讲授 | 用"时间维度的处理方式"这条线串联：CNN忽略时序→RNN建模时序→Attention聚焦关键帧 |
| 1:10-1:20 | 休息 | — | — |
| 1:20-1:50 | 搭建CRNN（CNN+GRU+FC）音频分类器 | 边讲边练 | 这是最实用的音频分类架构，也是理解DeepFilterNet中encoder-decoder的基础 |
| 1:50-2:20 | 训练与评估：混淆矩阵、精确率/召回率、F1 | 边讲边练 | 用ESC-50的10类声音做实验 |
| 2:20-2:30 | 总结：从通用分类到音频分类的关键差异 | 讲授 | 为下节课的CI特定任务做铺垫 |

**关键设计：** CRNN是音频分类的"workhorse"架构，同时也是理解更复杂模型的基础。DeepACE中的时频模式提取、DeepFilterNet中的编码器-解码器结构，都可以在CRNN中找到原型。把这节课的CRNN讲透了，后面的课事半功倍。

#### 第3次课：CI相关音频分类实战（2.5小时）

**教学目标：** 将通用音频分类方法应用到CI相关的研究问题上，完成主线项目的语音/噪声分类模块。

**内容与节奏：**

| 时间 | 内容 | 形式 | 要点 |
|------|------|------|------|
| 0:00-0:15 | 回顾 + 上节课任务答疑 | 互动 | — |
| 0:15-0:40 | CI特定问题：电刺激通道选择、语音活动检测(VAD)、环境声分类 | 讲授 | 让学生理解"分类"在CI研究中的具体应用场景 |
| 0:40-1:10 | 实战1：语音活动检测(VAD)模型 | 边讲边练 | 输入=音频帧，输出=语音/非语音。这跟ACE策略中的"选通道"逻辑直接对应 |
| 1:10-1:20 | 休息 | — | — |

**课时：2-3次课 × 2.5小时 = 5-7.5小时**

DeepFilterNet是面向实时语音增强的深度学习模型，其设计考虑了低延迟和计算效率，与CI/助听器场景天然契合。

## 三、培训形式设计

### 3.1 课堂形式

采用"15分钟讲授 + 10分钟实操 + 循环"的节奏，而不是传统的"先讲1小时再练1小时"。原因：

- 零基础学生的注意力集中时间约15-20分钟
- 知识的消化需要即时的动手验证
- 频繁的实操减少了"听完就忘"的问题
- 每个小实操都给出明确的预期输出，学生能即时验证自己是否做对了

### 3.2 课前准备（极其重要）

每次课前1-2天，分发以下材料：

- **预习文档**（1-2页）：本次课的核心概念、关键术语、思考题。不是课本章节，而是一份精心写的"你应该知道什么"。
- **环境检查清单**：确保学生的环境已经就绪。每次课都花5分钟确认环境，比课上花30分钟调试强100倍。
- **代码仓库更新**：所有课程代码放在一个Git仓库中，每次课前更新。学生只需 `git pull`。

### 3.3 课后支撑

- **答疑时间**：每周固定1小时的答疑时段（线上或线下），学生可以来问任何问题
- **内部文档Wiki**：用Notion或GitBook建立一个课程Wiki，记录每次课的要点、常见问题、代码片段
- **代码模板**：每个实操环节都提供"起步代码"（starter code），学生填空而不是从零写，降低挫败感

### 3.4 分层教学策略

每个实操环节和课后任务都设计两个层次：

| 层次 | 目标 | 完成标准 |
|------|------|----------|
| 基础 | 跟着步骤走，能跑通，能理解每一步在做什么 | 代码能运行，输出正确，能回答"这一步做了什么" |
| 进阶 | 在基础之上进行修改、扩展、对比实验 | 能提出修改方案，实施修改，分析结果差异 |

进阶任务不是"多做题"，而是"对同一件事做更深的思考"。这避免了基础弱的学生感到压力，也让基础强的学生不无聊。

建议安排在6-8周内完成，每周2次课：

| 周 | 课1 | 课2 | 备注 |
|----|-----|-----|------|
| 第1周 | 模块1-1：Linux基础 | 模块1-2：环境搭建 | 本周重点确保环境全部就绪 |
| 第2周 | 模块2-1：感知机与神经网络 | 模块2-2：CNN与语谱图 | — |
| 第3周 | 模块2-3：训练技巧 | 模块3-1：音频特征 | 模块2到3的过渡周 |
| 第4周 | 模块3-2：音频分类模型 | 模块3-3：CI分类实战 | — |
| 第5周 | 模块4-1：DeepACE论文 | 模块4-2：DeepACE代码 | 进入论文+代码模式 |
| 第6周 | 模块4-3：DeepACE实验（可选） | 模块5-1：语音增强+DeepFilterNet论文 | — |
| 第7周 | 模块5-2：DeepFilterNet代码 | 模块5-3：DeepFilterNet+CI实验（可选） | — |
| 第8周 | 模块6-1：ASR原理 | 模块6-2：ASR实战+Pipeline整合 | 最终整合 |

### 4.2 弹性空间

- 模块4和5的第3次课标记为"可选"，可根据学生的进度灵活安排。如果前两次课后学生还有疑问或需要更多实操时间，就上第3次课；否则可以直接进入下一个模块。
- 如果整体进度落后，可以压缩模块1（如果学生Linux基础尚可）或模块6（ASR简介不需要太深）。
- 每个模块结束后留1-2天的消化时间，不要连续密集排课。

## 五、评估与反馈

### 5.1 过程性评估

不采用传统考试，而是以下方式：

- **每次课的实操完成度**：能否在课上完成实操任务？是否需要额外帮助？
- **课后任务提交**：代码+简短报告（1页以内）
- **代码Review**：挑选学生的代码在下次课开始时进行简短review（匿名），点评常见的代码习惯问题

### 5.2 终结性评估

培训结束时的评估方式：

- **Pipeline演示**：每个学生（或2人小组）展示自己的完整pipeline运行结果
- **技术报告**：写一份3-5页的技术报告，内容包括pipeline各模块的作用、关键参数选择、实验结果分析
- **代码提交**：将所有代码提交到课程Git仓库

### 5.3 反馈机制

- **课后即时反馈**：每次课结束前2分钟，让学生写一个"今天最清楚的一点 + 今天最模糊的一点"
- **中期反馈**：模块3结束后进行一次匿名问卷，调整后续课程的节奏和深度
- **同伴学习**：鼓励学生之间讨论和互助，设置一个课程群用于问答

## 六、常见风险与应对

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|----------|
| 环境配置问题导致大量时间浪费 | 高 | 严重 | 预配置Docker镜像+详细文档+课前环境检查；课上配1-2名助教 |
| 学生编程基础差距大，进度难以统一 | 高 | 中等 | 分层任务+同伴互助；基础弱的学生降低课后任务要求 |

## 八、给培训者的建议

### 8.1 心态调整

你是一个研究者，现在要当老师。这两个角色的思维方式不同：

- 研究者追求"深入"，教师追求"理解"。你花了两年理解的东西，不可能在2.5小时内让学生完全理解。要接受"讲80%，学生吸收60%，记住40%，能用20%"这个现实——20%已经很好了。
- 研究者习惯"先证明再接受"，教师应该"先接受再理解"。让学生先用起来，产生好奇心，再讲为什么。
- 你会觉得很多东西"这么简单为什么不懂"——这是知识的诅咒。解决办法：回忆你第一次学这些东西的时候卡在了哪里，把那个卡点讲清楚。

### 8.2 课堂管理

- **每15-20分钟切换一次活动**（讲授→实操→讨论），保持节奏感
- **实操环节不要等所有人完成**——设定一个时间限制，到时间就展示参考解法，没完成的学生课后补
- **鼓励提问但不要被提问带跑**——把复杂问题留到课后或答疑时间
- **用屏幕共享/投影展示预期输出**，让学生知道自己做对了没有

### 8.3 持续改进

- **每次课后自己写5分钟复盘**：什么讲得好？什么讲得急？学生卡在哪里？下次怎么改？
- **让学生教学生**——最有学习效果的环节是让学生给其他同学讲解。培训后半段可以安排学生做5分钟的知识分享。
- **积累FAQ**——每个常见问题都记下来，下次课直接作为预习材料分发。
